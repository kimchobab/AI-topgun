## 파일별 상세 가이드

### `my_reward.py` ★★★

- **무엇을 바꾸는가**: 매 스텝 보상값과 에피소드 종료 시 터미널 보상 설계
- **잘못 바꾸면**: 반환 형식이 `(float, dict)`가 아니면 훈련 루프 즉시 crash. `dict` 키 이름은 자유롭게 정해도 됨
- **최소 제출 기준**: 기본 제공 `reward.py`를 그대로 써도 제출 가능. 단, 성능 차별화는 여기서 결정됨

### `my_train.py` ★★★

- **무엇을 바꾸는가**: 알고리즘(SAC/PPO), 하이퍼파라미터, `observation_mode`, `reward_module` 연결
- **잘못 바꾸면**: `observation_mode`를 `my_submission.py`와 다르게 설정하면 제출 후 전혀 다른 행동 발생
- **최소 제출 기준**: `observation_mode`, `reward_module` 경로, 출력 번들 경로 반드시 명시

### `my_submission.py` ★★★

- **무엇을 바꾸는가**: 제출용 정책 로딩, `observation_mode`, `ACTION_REPEAT` 설정
- **잘못 바꾸면**: 훈련과 `observation_mode` 또는 `ACTION_REPEAT`이 다르면 제출 결과가 훈련 결과와 완전히 달라짐

> ⚠️ **제출 전 필수 3가지 확인**
> 1. `observation_mode` — `my_train.py`와 동일한가?
> 2. `ACTION_REPEAT` — `my_train.py`의 `step_ratio`와 동일한가?
> 3. `bundle_dir` — 실제 저장된 체크포인트 경로가 맞는가?

### `my_observation.py` ★★☆

- **무엇을 바꾸는가**: 커스텀 관측 벡터 설계 (`OBSERVATION_SIZE`, `build_observation` 함수)
- **잘못 바꾸면**: `OBSERVATION_SIZE`와 실제 반환 배열 크기가 다르면 Gym space 불일치로 crash
- **최소 제출 기준**: 사용 시 `my_train.py`와 `my_submission.py` 모두 `--observation-module`로 동일하게 지정할 것

### `my_curriculum.py` ★☆☆

- **무엇을 바꾸는가**: 커리큘럼 단계 정의, 진급 조건, 각 단계별 보상 오버라이드
- **잘못 바꾸면**: 진급 조건이 너무 낮으면 조기 진급으로 충분한 학습 없이 다음 단계 진입
- **최소 제출 기준**: 기본 `curriculum.py` 사용 가능. 커스텀 시 `train_curriculum.py --stages-module student.my_curriculum` 으로 실행

### `Rule_팀이름.xml` ★☆☆

- **무엇을 바꾸는가**: BT 상대방 행동 규칙 (Hybrid/BT 모드에서만 필요)
- **잘못 바꾸면**: XML 문법 오류 시 DLL 로드 실패. 알 수 없는 태스크 이름 사용 시 묵시적 무시
- **최소 제출 기준**: RL 단독 제출 시 불필요. Hybrid 모드 제출 시만 필요
