# 16 - 피드백

## 전체 총평

`00`부터 `14`까지의 문서는 전체적으로 구조가 잘 잡혀 있다.  
특히 아래 강점이 뚜렷하다.

- 학습 파이프라인을 `환경 → 관측 → 행동 → 데미지 → 보상 → 종료 → 훈련` 흐름으로 단계화해 설명한 점이 좋다.
- 학생이 실제로 손대야 하는 지점을 `reward / observation / curriculum / submission / BT rule`로 분리해 보여줘 진입 장벽을 낮춘다.
- `WEZ`, `ATA`, `Pursuit shaping`, `Hybrid residual` 같은 핵심 개념을 문서 여러 곳에서 반복해 강조한 점도 교육용으로 효과적이다.

다만, 실제 사용 관점에서는 몇 가지 **문서 간 불일치**, **정의 충돌**, **초심자에게 빠진 전제 설명**이 있어서 이 부분만 정리되면 훨씬 강한 자료가 된다.

---

## 가장 먼저 고치면 좋은 것

### 1. 문서 간 설정값 불일치 정리 필요

가장 큰 문제는 문서마다 “기본값”과 “권장값”이 섞여 있어 읽는 사람이 혼동하기 쉽다는 점이다.

- `02 - 환경 아키텍처.md` 에서는 `observation_mode = "classic12"` 가 기본값으로 적혀 있다.
- `04 - 관측 공간.md`, `12 - 강화학습 훈련.md`, `13 - 실행 가이드.md` 에서는 사실상 `tactical16` 을 기본처럼 다룬다.

이 경우 아래처럼 구분해 써주는 게 좋다.

- 코드 기본값: `classic12`
- 실습/권장값: `tactical16`

지금 상태에서는 “실제로 아무 설정 없이 돌리면 무엇이 선택되는가”가 한 번에 안 보인다.

### 2. 커리큘럼 설명이 `01` 과 `12` 사이에서 다름

`01 - 파이프라인 전체 흐름.md` 의 커리큘럼 그림은:

- Stage0: Loiter 상대
- Stage1: BT 기본 상대
- Stage2: BT 고급 상대
- Stage3: Self-play

하지만 `12 - 강화학습 훈련.md` 의 실제 기준은:

- Stage0: Fixed 고정 타겟
- Stage1: Fixed 고정 타겟
- Stage2: Loiter
- Stage3: Autopilot
- Stage4~13: Two-Circle Headon
- Stage14: Full dogfight

이건 단순 요약 차원을 넘어서 **학습 단계 정의가 다르게 보이는 수준**이라 반드시 맞춰야 한다.  
가장 좋은 방식은 `01`의 다이어그램 옆에 “개념 요약 그림”이라고 명시하거나, 실제 stage 구조를 반영한 축약 그림으로 바꾸는 것이다.

### 3. AA(Aspect Angle) 설명이 내부적으로 충돌함

`11 - 기하학 계산.md` 안에서 Aspect Angle 설명이 서로 다르게 읽힌다.

- 전술 표에서는 `AA = 0° (6시) = 적기 바로 뒤` 라고 설명한다.
- 아래 `Aspect Angle (AA)` 코드 설명에서는 `AA=0° → 아군이 적기 정면`, `AA=180° → 아군이 적기 뒤` 라고 적혀 있다.

이건 같은 문서 안의 직접 충돌이다.  
AA 정의를 코드 기준으로 하나로 고정하고, 그림과 표를 그 정의에 맞춰 전부 다시 맞추는 게 필요하다.

### 4. WEZ 조건 표기 일관성 필요

`08 - WEZ 모델.md` 안에서도 조건 표기가 조금씩 다르다.

- 본문: `ATA < 1° AND 152m ≤ 거리 ≤ 914m`
- 계산식: `if min_range <= distance <= max_range` 와 `|ATA| <= angle_deg/2`
- 시각화 문구: `ATA < 1° AND 152m < dist < 914m`

`<` 인지 `<=` 인지 문서가 통일되어야 한다.  
실제 코드 기준으로 하나만 남겨야 한다.

### 5. 종료 조건 표에서 `terminated/truncated` 표기가 헷갈림

`07 - 종료 조건.md` 의 outcome 매핑 표에서:

- `max time out`
- `episode step limit`

이 둘은 설명상 `truncated` 인데, 표에서는 `terminated` 열 위치가 애매하게 보인다.  
표 열을 `terminated / truncated / outcome` 으로 분리해서 명확하게 다시 쓰는 편이 좋다.

## 문서별 상세 피드백

### 00 - 전체 인덱스

좋은 점:

- 전체 시스템 구조와 핵심 파일 경로를 빠르게 파악하기 좋다.
- 처음 보는 사람도 어떤 문서를 어떤 순서로 읽을지 감이 잡힌다.

보완점:

- “처음 읽는 순서 추천” 섹션이 있으면 더 좋다.
- 예: `00 → 01 → 02 → 04 → 05 → 06 → 07 → 12 → 13 → 14`

### 01 - 파이프라인 전체 흐름

좋은 점:

- episode loop 와 training loop 를 나눠 보여준 점이 좋다.
- 환경과 학습이 어떻게 연결되는지 직관적으로 이해된다.

보완점:

- 앞서 말한 커리큘럼 도식 불일치를 수정해야 한다.
- `env.step(action)` 이후 보상과 종료가 계산되는 순서가 코드와 정말 일치하는지 한 줄로 명시하면 더 좋다.
- “학생이 실제로 개입하는 지점”을 루프 그림에 색이나 주석으로 표시하면 교육 효과가 올라간다.

### 02 - 환경 아키텍처

좋은 점:

- 생성자 파라미터와 `step()` 반환값 설명이 실용적이다.

보완점:

- `DogFightEnv` 와 `DogFightWrapper` 관계를 좀 더 명확히 써주면 좋다.
- `DEFAULT_ENV_CONFIG` 와 실습 권장 설정을 분리해야 한다.
- `target_mode` 와 `target_action_provider` 가 동시에 등장하는데, 둘의 우선순위가 어떻게 되는지 설명이 있으면 좋다.

### 03 - 상태 스키마

좋은 점:

- 최소한 자주 쓰는 인덱스를 추려 보여준 점이 좋다.

보완점:

- “46개 중 실제 실습에서 자주 쓰는 것 10개” 식의 별도 요약이 있으면 더 빠르다.
- `ALT` 와 `D` 가 둘 다 고도 관련 값인데 어떤 상황에서 무엇을 써야 하는지 구분 설명이 있으면 좋다.

### 04 - 관측 공간

좋은 점:

- `tactical16` 를 사실상 표준 관측으로 잘 정리했다.

보완점:

- `relative14` 의 정규화 범위 표기 `[-10000/5000, +10000/5000]` 는 처음 보면 수식인지 범위인지 혼동된다.
- 각 피처가 실제로 `값 / 최대범위` 로 정규화되는지, 아니면 단순 허용범위만 적은 건지 명확히 써야 한다.
- `tactical16` 예시 벡터 1개가 있으면 훨씬 이해가 쉬워진다.

### 05 - 행동 공간

좋은 점:

- throttle 변환을 강조한 건 매우 중요하고 잘했다.

보완점:

- action repeat 또는 제어 주기와 행동 적용 주기의 관계가 여기에도 한 줄 들어가면 좋다.
- 스로틀 0.5가 “실속 방지용 중립”이라는 설명은 좋은데, 실제로 어느 정도 속도를 유지하는지는 감각적으로 한 번 더 설명해주면 좋다.

### 06 - 보상 함수

좋은 점:

- shaping 보상과 terminal reward 의 역할 분리가 명확하다.

보완점:

- 인덱스 문서에서는 “5가지 보상 컴포넌트”라고 했는데 실제 문서에는 `survival, step, pursuit, damage, safety, terminal` 총 6개가 나온다.
- `truncated=True` 일 때 terminal reward 를 어떻게 처리하는지 예시가 있으면 더 좋다.
- 학생 입장에서는 “처음 바꿔볼 만한 파라미터 3개”를 따로 박스 처리해주면 실습성이 더 올라간다.

### 07 - 종료 조건

좋은 점:

- 종료 우선순위를 써준 건 디버깅에 매우 유용하다.

보완점:

- `fuel fail` 과 `time out` 이 동시에 가까워질 때 어떤 조건이 먼저 잡히는지 예시가 있으면 좋다.
- `Two-Circle guard` 는 개념상 어려워서, 작은 그림이나 사례가 하나 더 있으면 이해가 빨라진다.

### 08 - WEZ 모델

좋은 점:

- 실제 데미지 메커니즘을 매우 명확하게 설명했다.
- shaping 과 실탄 판정의 간극을 짚은 부분이 좋다.

보완점:

- 아군 기준 `in_wez` 와 적기 기준 `enemy_in_wez` 를 구분한 보조 설명이 있으면 좋다.
- 지금은 “내가 때리고 있나”만 보이는데, “내가 맞고 있나” 플래그도 학생이 이해하기 쉽게 같이 언급하면 좋다.

### 09 - 행동 제공자

좋은 점:

- BT, RL, Hybrid 역할 차이를 빠르게 이해할 수 있다.

보완점:

- `primary_provider=rl`, `secondary_provider=bt` 로 예시를 들면서 계산식은 `BT + residual_scale * RL` 로 적혀 있어 읽는 사람이 “누가 기준이고 누가 잔차인가”를 헷갈릴 수 있다.
- residual 모드에서 최종 action clipping 이 어디서 일어나는지도 한 줄 있으면 좋다.

### 10 - 비헤이비어 트리

좋은 점:

- BT 내부 개념을 Python 사용자도 이해할 수 있게 번역해놓은 점이 좋다.

보완점:

- `Task_Empty` 만 보면 실제로는 기동이 거의 없는 것처럼 보일 수 있다.
- “실제 기동 태스크는 DLL 내부에 더 있으며, 문서에서는 최소 예시만 든다”는 안내가 있으면 오해를 줄일 수 있다.
- `Rule.xml` 수정 시 어떤 변경이 안전하고 어떤 변경이 위험한지도 짧게 써주면 좋다.

### 11 - 기하학 계산

좋은 점:

- 전투 기하 개념을 문서화한 건 매우 가치가 크다.

보완점:

- 가장 시급한 건 AA 정의 충돌 수정이다.
- 회전행렬 순서 설명과 주석이 직관적이지만, 실제 코드와 동일한 축 순서인지 한 번 더 맞춰야 한다.
- `proj=True/False` 차이에 대해 “보상은 무엇을 쓰고, WEZ는 무엇을 쓰며, 관측은 무엇을 쓰는가”를 표로 묶으면 더 좋다.

### 12 - 강화학습 훈련

좋은 점:

- 이 문서는 실전성이 높다.
- stage 구조, 진급 조건, 보상 오버라이드, 하이퍼파라미터가 핵심만 잘 모여 있다.

보완점:

- `01`과 커리큘럼 설명을 통일해야 한다.
- `iterations=50` 이 예시값인지 권장 시작점인지 명확히 하면 좋다.
- 체크포인트를 어떤 기준으로 “쓸 만한 모델”로 판단할지 평가 기준이 있으면 좋다.

### 13 - 실행 가이드

좋은 점:

- 바로 실행 가능한 예시가 많아서 실습용으로 좋다.

보완점:

- 경로가 예시 경로인지 실제 권장 경로인지 명시가 필요하다.
- `step_ratio` 와 `ACTION_REPEAT` 중요도는 매우 큰데, 이건 실행 가이드뿐 아니라 `12`, `14`에도 중복 강조하는 편이 좋다.
- 실행 전 체크리스트 5줄 버전이 있으면 초심자 실수가 크게 줄어든다.

### 14 - 학생 구현 포인트

좋은 점:

- 우선순위를 명시한 건 매우 좋다.

보완점:

- 지금은 파일명만 있어 다소 건조하다.
- 각 파일마다 “무엇을 바꾸는 파일인지”, “잘못 바꾸면 어떤 문제가 나는지”, “최소 제출 기준이 무엇인지”를 한 줄씩 추가하면 훨씬 강해진다.
- 특히 `my_submission.py` 는 실전 제출과 직결되므로 `observation_mode`, `ACTION_REPEAT`, bundle path/weight 로딩 체크를 별도 경고로 빼는 게 좋다.

## 학생 관점에서 추가되면 좋은 것

### 1. “최소 성공 경로” 문서가 있으면 좋음

현재 문서는 정보량이 충분하지만, 처음 하는 학생은 오히려 어디부터 건드릴지 헷갈릴 수 있다.  
아래 같은 1페이지짜리 최소 경로가 있으면 좋다.

1. `tactical16` 고정
2. 기본 보상으로 먼저 학습
3. `my_reward.py` 에서 `step_penalty`, `pursuit_scale`, `win_reward` 먼저 조절
4. `run_local_dogfight.py` 로 RL vs BT 테스트
5. `my_submission.py` 에 observation/action 설정 동기화

### 2. 실패 사례 중심 설명이 더 있으면 좋음

예를 들어 아래는 학생이 자주 틀릴 포인트다.

- observation_mode 불일치
- action repeat 불일치
- throttle 해석 실수
- WEZ 는 못 들어가는데 pursuit reward 만 높아지는 경우
- stage 진급 조건을 못 맞추는 경우

이런 “왜 안 되는가” 중심 예시가 들어가면 자료 가치가 더 올라간다.

### 3. 용어집이 있으면 좋음

아래 용어는 처음 보는 사람에게 어렵다.

- ATA
- AA
- HCA
- LOS
- WEZ
- OBFM / DBFM / HABFM
- residual policy

지금은 각 문서에 흩어져 있어서, 별도 1페이지 용어집이 있으면 탐색 비용이 줄어든다.

## 최종 한 줄 피드백

현재 문서는 **구조와 핵심 개념은 상당히 잘 잡혀 있지만, 실제 실습에서 바로 쓰기 위해서는 문서 간 정의와 기본값을 먼저 통일해야 한다.**  
특히 `커리큘럼`, `AA 정의`, `WEZ 조건`, `observation 기본값` 네 군데만 먼저 정리해도 전체 문서 신뢰도가 크게 올라간다.
